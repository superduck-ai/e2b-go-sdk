# Connect LLMs to E2B

The Go SDK in this repository does not include a built-in model client or a notebook-style `runCode(...)` helper.

Instead, connect any LLM by exposing a small set of sandbox tools from your Go application and letting the model call them.

The usual loop is:

1. your provider SDK gets a user message
2. the model returns a tool name plus JSON arguments
3. your Go code dispatches that tool call to `sandbox.Commands.Run(...)`, `sandbox.Files.Write(...)`, `sandbox.Files.Read(...)`, or other sandbox APIs
4. you send the tool result back to the model

This pattern works with hosted providers and orchestration frameworks alike. The E2B-specific part is the tool dispatcher.

## Expose command execution as a tool

This is the simplest building block when the model needs shell access inside a sandbox.

```go
package main

import (
	"context"
	"encoding/json"
	"fmt"
	"log"

	e2b "github.com/superduck-ai/e2b-go-sdk"
)

type ToolCall struct {
	Name      string
	Arguments json.RawMessage
}

type ExecuteCommandArgs struct {
	Command string `json:"command"`
	Cwd     string `json:"cwd,omitempty"`
}

func dispatchTool(ctx context.Context, sandbox *e2b.Sandbox, call ToolCall) (string, error) {
	switch call.Name {
	case "execute_command":
		var args ExecuteCommandArgs
		if err := json.Unmarshal(call.Arguments, &args); err != nil {
			return "", err
		}

		execution, err := sandbox.Commands.Run(ctx, args.Command, &e2b.CommandStartOpts{
			Cwd: args.Cwd,
		})
		if err != nil {
			return "", err
		}
		result := execution.(*e2b.CommandResult)
		return result.Stdout, nil
	default:
		return "", fmt.Errorf("unknown tool %q", call.Name)
	}
}

func main() {
	ctx := context.Background()

	sandbox, err := e2b.Create(ctx, "base", nil)
	if err != nil {
		log.Fatal(err)
	}
	defer sandbox.Kill(context.Background(), nil)

	call := ToolCall{
		Name:      "execute_command",
		Arguments: json.RawMessage(`{"command":"uname -a","cwd":"/home/user"}`),
	}

	output, err := dispatchTool(ctx, sandbox, call)
	if err != nil {
		log.Fatal(err)
	}

	log.Println(output)
}
```

## Add file tools

Most agents also need to write prompts, save intermediate files, and read results back out of the sandbox.

```go
package main

import (
	"context"
	"encoding/json"
	"fmt"
	"log"

	e2b "github.com/superduck-ai/e2b-go-sdk"
)

type ToolCall struct {
	Name      string
	Arguments json.RawMessage
}

type WriteFileArgs struct {
	Path    string `json:"path"`
	Content string `json:"content"`
}

type ReadFileArgs struct {
	Path string `json:"path"`
}

func dispatchFileTool(ctx context.Context, sandbox *e2b.Sandbox, call ToolCall) (string, error) {
	switch call.Name {
	case "write_file":
		var args WriteFileArgs
		if err := json.Unmarshal(call.Arguments, &args); err != nil {
			return "", err
		}

		if _, err := sandbox.Files.Write(ctx, args.Path, args.Content, nil); err != nil {
			return "", err
		}
		return "ok", nil
	case "read_file":
		var args ReadFileArgs
		if err := json.Unmarshal(call.Arguments, &args); err != nil {
			return "", err
		}

		value, err := sandbox.Files.Read(ctx, args.Path, nil)
		if err != nil {
			return "", err
		}
		return value.(string), nil
	default:
		return "", fmt.Errorf("unknown tool %q", call.Name)
	}
}

func main() {
	ctx := context.Background()

	sandbox, err := e2b.Create(ctx, "base", nil)
	if err != nil {
		log.Fatal(err)
	}
	defer sandbox.Kill(context.Background(), nil)

	_, err = dispatchFileTool(ctx, sandbox, ToolCall{
		Name:      "write_file",
		Arguments: json.RawMessage(`{"path":"/home/user/task.txt","content":"Summarize the repository"}`),
	})
	if err != nil {
		log.Fatal(err)
	}

	content, err := dispatchFileTool(ctx, sandbox, ToolCall{
		Name:      "read_file",
		Arguments: json.RawMessage(`{"path":"/home/user/task.txt"}`),
	})
	if err != nil {
		log.Fatal(err)
	}

	log.Println(content)
}
```

If the model needs batch writes, use `sandbox.Files.WriteFiles(...)`.

## Reuse one sandbox per conversation or thread

For stateful agents, store `sandbox.SandboxID` alongside the conversation ID in your own application state. Then reconnect on the next tool call instead of creating a fresh sandbox every turn.

```go
package main

import (
	"context"
	"log"

	e2b "github.com/superduck-ai/e2b-go-sdk"
)

type Conversation struct {
	SandboxID string
}

func main() {
	ctx := context.Background()
	timeoutMs := 600_000

	sandbox, err := e2b.Create(ctx, "base", &e2b.SandboxOpts{
		TimeoutMs: &timeoutMs,
		Lifecycle: &e2b.SandboxLifecycle{
			OnTimeout: "pause",
		},
	})
	if err != nil {
		log.Fatal(err)
	}

	conversation := Conversation{
		SandboxID: sandbox.SandboxID,
	}

	paused, err := sandbox.Pause(ctx, nil)
	if err != nil {
		log.Fatal(err)
	}

	resumed, err := e2b.Connect(ctx, conversation.SandboxID, nil)
	if err != nil {
		log.Fatal(err)
	}

	execution, err := resumed.Commands.Run(ctx, "pwd", nil)
	if err != nil {
		log.Fatal(err)
	}
	result := execution.(*e2b.CommandResult)

	log.Println(paused, result.Stdout)
}
```

Pause or kill the sandbox when the conversation is idle, depending on whether you want to preserve filesystem state.

## Provider and framework notes

OpenAI, Anthropic, Mistral, Groq, LangChain, LlamaIndex, CrewAI, Ollama, and similar frameworks all reduce to the same E2B integration shape:

- define a tool schema in the provider or orchestration layer
- deserialize the model's tool arguments in Go
- call the matching sandbox API
- send the result back to the model

If your model cannot call tools, the fallback is to ask it to emit explicit shell commands or source code, validate that output in your app, and then run it through `sandbox.Commands.Run(...)` or a preinstalled interpreter inside the sandbox.

For longer-running agent workflows, see [Coding Agents](/docs/use-cases/coding-agents) and [Computer Use](/docs/use-cases/computer-use).
